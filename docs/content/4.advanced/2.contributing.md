# Contributing

:ellipsis{right=0px width=50% blur=100px}

We welcome contributions! Here's how to get involved.

::alert{type="success"}
**Issue-First Development.** Every change starts with an issue. This keeps work organized and avoids duplicate effort.
::

## Getting Started

::steps
### Fork and Clone

```bash
git clone https://github.com/YOUR_USERNAME/capydeploy
cd capydeploy
```

### Initialize Submodules

```bash
git submodule update --init --recursive
```

### Install Dependencies

::list{type="info"}
- **Go 1.24+** — Backend language
- **Wails CLI** — Desktop app framework
- **Bun** — Frontend package manager
::

### Run in Dev Mode

::code-group
```bash [Hub]
cd apps/hub
wails dev
```

```bash [Agent]
cd apps/agent
wails dev
```
::
::

## Development Workflow

::steps
### Find or Create an Issue

Before writing code, make sure there's an issue describing the work. Check [existing issues](https://github.com/lobinuxsoft/capydeploy/issues) first.

### Create a Branch

Use GitHub CLI (recommended):
```bash
gh issue develop <NUMBER> --base development --checkout
```

Or manually:
```bash
git checkout development
git checkout -b feat/issue-123-short-description
```

### Make Your Changes

::list{type="success"}
- Write clean, tested code
- Follow the code standards below
- Update docs if needed
::

### Commit

```bash
git commit -m "feat(agent): agregar soporte macOS"
```

### Create a Pull Request

```bash
gh pr create --base development --title "feat: short description" --body "Closes #123"
```

### Wait for Review

Don't merge your own PRs. A maintainer will review and merge.
::

## Code Standards

### Naming Conventions

| Language | Classes | Functions | Private |
|----------|---------|-----------|---------|
| Go | PascalCase | camelCase | lowercase |
| TypeScript | PascalCase | camelCase | _prefix |

::alert{type="info"}
**Files:** Always `snake_case` for file names in all languages.
::

### Guidelines

::card-grid
  ::card{icon=ph:book-open-duotone}
  #title
  Document APIs
  #description
  Public functions need GoDoc comments. Exported types need descriptions.
  ::

  ::card{icon=ph:warning-duotone}
  #title
  Wrap Errors
  #description
  Always add context: `fmt.Errorf("failed to X: %w", err)`
  ::

  ::card{icon=ph:lock-duotone}
  #title
  No Secrets
  #description
  Never log passwords, tokens, or API keys. Ever.
  ::

  ::card{icon=ph:test-tube-duotone}
  #title
  Write Tests
  #description
  New features need tests. Bug fixes need regression tests.
  ::
::

### Principles

::list{type="primary"}
- **SOLID** — Single responsibility, open/closed, etc.
- **Composition > Inheritance** — Embed, don't extend
- **KISS** — Keep it simple, avoid over-engineering
- **YAGNI** — Don't add features "just in case"
::

## Commit Messages

We use **Conventional Commits** in **Spanish**:

::code-group
```bash [Features]
git commit -m "feat(agent): agregar soporte para macOS"
git commit -m "feat(hub): implementar búsqueda de artwork"
```

```bash [Bug Fixes]
git commit -m "fix(hub): corregir descubrimiento mDNS"
git commit -m "fix(agent): resolver timeout en uploads grandes"
```

```bash [Other]
git commit -m "docs: actualizar guía de instalación"
git commit -m "chore: actualizar dependencias"
git commit -m "refactor(protocol): simplificar tipos de mensaje"
```
::

::alert{type="danger"}
**No AI signatures** in commits or PRs. No "Generated with..." or "Co-Authored-By: AI...".
::

## Branch Strategy

| Branch | Purpose | Base |
|--------|---------|------|
| `main` | Stable releases | — |
| `development` | Integration branch | `main` |
| `feat/*` | New features | `development` |
| `fix/*` | Bug fixes | `development` |

::alert{type="warning"}
**Never push directly to `main`.** All changes go through `development` via PR.
::

## Building

Always use the project's build scripts:

::code-group
```bash [Hub]
cd apps/hub
./build.sh        # Linux/macOS
.\build.bat       # Windows
```

```bash [Agent]
cd apps/agent
./build.sh        # Linux/macOS
.\build.bat       # Windows
```
::

## Testing

```bash
# Run all tests
go test ./...

# With coverage
go test -cover ./...

# Specific package
go test ./pkg/protocol/...

# Verbose output
go test -v ./pkg/transfer/...
```
